# 相对位置融合模块：架构与算法落地说明

本文档从“当前仓库真实条件”出发，重新梳理 UAV–UGV 相对平面位置融合模块的架构与算法。
它不是论文推导的纯数学复述，而是**可直接指导工程实现的架构说明**。

---

## 1. 当前仓库中的真实前提

在进入算法之前，先明确这个模块要运行在什么环境里。

### 1.1 坐标系前提

当前仓库已经约定了以下坐标层次：

- `global`：联合系统共享的逻辑全局参考系；
- `ugv_map`：UGV 侧地图参考系；
- `uav_map`：UAV 侧地图参考系；
- `ugv_odom`、`uav_odom`：各自本地里程系；
- `ugv_base_footprint`、`uav_base_link`：各自机体基座系。

当前联合仿真通过静态 TF 把 `global -> ugv_map`、`global -> uav_map` 连起来；
当前 UAV 仿真还明确使用了 `uav_map -> uav_odom -> uav_base_link` 的层级契约。

**本模块的核心假设是：**

- 所有输入都必须先被适配或变换到统一的逻辑 `global` 坐标后，才能进入滤波核心；
- 核心滤波器内部不直接关心 `uav_map`、`ugv_map`、`odom` 等本地坐标系；
- 当前仓库里的“动态全局对齐”尚未实现，因此 v1 不把坐标对齐问题继续下推给滤波器。

### 1.2 各端可获得的数据

#### UAV 侧

- 仿真：`/uav/odom` 已存在；
- 真机：可通过 `px4_planar_state_reader_node` 产出平面 `nav_msgs/Odometry`；
- 可获得：
  - 全局平面位置 `p_a = [x_a, y_a]^T`
  - 全局平面速度 `v_a = [v_ax, v_ay]^T`
  - 位置协方差的 `x/y` 子块 `Σ_a ∈ R^{2×2}`

#### UGV 侧

- 真机与仿真通用：`/ugv/odom`
- 导航场景可额外获得：
  - `/ugv/odometry/filtered`
  - `/amcl_pose`
- 可获得：
  - 全局平面位置 `p_g = [x_g, y_g]^T`
  - 全局平面速度 `v_g = [v_gx, v_gy]^T`
  - 位置协方差 `Σ_g ∈ R^{2×2}`

### 1.3 v1 要解决的核心问题

在任一离散时刻，需要输出：

- UGV 相对 UAV 的二维位置估计；
- 该估计的二维协方差；
- 一个可用于上层策略判断的保守误差上界；
- 一个“需要执行外部重定位”的触发信号。

这里的“外部重定位”不是模块自己完成，而是**模块只负责发出事件信号**。

## 2. 为什么 v1 只做二维 `xy`

当前版本固定只估计相对平面位置 `r ∈ R²`，不估计相对航向 `ψ`，也不在状态中显式估计相对速度。

原因有三点：

1. **当前任务目标主要是平面会合 / 跟随 / 进入视觉降落前的粗定位。**
   真正末端对接的厘米级误差由 AprilTag / 视觉降落链路接管，没必要让相对位置融合模块承担全部姿态闭环。

2. **当前输入里最稳定的是平面位置与平面速度。**
   UAV 与 UGV 的 yaw 来源不完全对称，且 UGV 全局 yaw 与局部速度方向并不总是同质量。v1 如果把相对 yaw 放进状态，会显著增加建模与调参复杂度。

3. **二维状态最容易在仿真和真机上共用。**
   只估计 `xy` 时，输入接口、触发逻辑、协方差解释都更稳定，便于先把整个链路搭起来。

所以，v1 的目标是：

- 用一个最小但完整的二维模型，把“相对平面位置估计 + 不确定性上界 + 重定位触发”做好；
- 把更高维的相对姿态问题留给后续版本。

## 3. 统一符号、来源、含义与形状

下面统一全部符号，避免 `v` 既表示速度又表示噪声的冲突。

### 3.1 时间与坐标

- `G`：逻辑全局坐标系，对应模块内部统一使用的 `global_frame`
- `k`：离散时刻索引
- `t_k`：第 `k` 次滤波 tick 的时间戳
- `Δt_k = t_k - t_{k-1}`：相邻 tick 的时间间隔

### 3.2 UAV / UGV 的输入状态

#### UAV

- `p_{a,k} ∈ R²`：UAV 在 `G` 中的平面位置，形状 `2×1`
- `v_{a,k} ∈ R²`：UAV 在 `G` 中的平面速度，形状 `2×1`
- `Σ_{a,k} ∈ R^{2×2}`：UAV 平面位置协方差，形状 `2×2`

#### UGV

- `p_{g,k} ∈ R²`：UGV 在 `G` 中的平面位置，形状 `2×1`
- `v_{g,k} ∈ R²`：UGV 在 `G` 中的平面速度，形状 `2×1`
- `Σ_{g,k} ∈ R^{2×2}`：UGV 平面位置协方差，形状 `2×2`

### 3.3 要估计的状态

定义 UGV 相对 UAV 的平面位置：

`r_k = p_{g,k} - p_{a,k} ∈ R²`

v1 只把 `r_k` 作为滤波状态，不把相对速度并入状态。

### 3.4 过程输入

定义相对速度输入：

`u_k = v_{g,k} - v_{a,k} ∈ R²`

这是已知输入，不是状态变量。

### 3.5 测量量

定义相对位置测量：

`z_k = p_{g,k} - p_{a,k} ∈ R²`

### 3.6 噪声与协方差

- `w_k ∈ R²`：过程噪声，表示预测过程中未建模误差的等效位置扰动
- `Q_k ∈ R^{2×2}`：过程噪声协方差
- `n_k ∈ R²`：测量噪声
- `R_k ∈ R^{2×2}`：测量噪声协方差

### 3.7 滤波内部量

- `r̂_{k|k-1} ∈ R²`：预测状态
- `r̂_{k|k} ∈ R²`：更新后状态
- `P_{k|k-1} ∈ R^{2×2}`：预测协方差
- `P_{k|k} ∈ R^{2×2}`：更新后协方差
- `y_k ∈ R²`：innovation / 残差
- `S_k ∈ R^{2×2}`：innovation covariance
- `K_k ∈ R^{2×2}`：Kalman 增益
- `d_k² ∈ R`：门控用马氏距离
- `ρ_k ∈ R`：保守误差半径，单位 m

## 4. 最小二维 KF 模型

### 4.1 过程模型

离散化后，状态模型写成：

`r_{k+1} = r_k + Δt_k * u_k + w_k`

其中：

- `u_k = v_{g,k} - v_{a,k}`
- `w_k ~ N(0, Q_k)`

v1 固定采用各向同性的连续时间扩散率模型：

`Q_k = q_rate(mode_k) * Δt_k * I_2`

这里：

- `q_rate(mode_k)` 的单位是 `m²/s`
- `Q_k` 的单位是 `m²`
- `mode_k` 由当前数据质量决定

可选模式固定为四种：

- `nominal`
- `measurement_missing`
- `gate_rejected`
- `source_degraded`

### 4.2 为什么 `Q_k` 这样写

这一步是本模块里最重要的工程化取舍之一。

如果写成固定每步增量：

`Q_k = q_const * I_2`

那么滤波频率一改，协方差增长速度就变了，仿真与真机的参数无法复用。

因此 v1 固定写成：

`Q_k = q_rate * Δt_k * I_2`

这样：

- 20 Hz、50 Hz、100 Hz 下的物理含义一致；
- 更适合通过“误差上界多长时间从 0.3 m 长到 1.0 m”来反标定参数；
- 更适合仿真与真机共用。

### 4.3 观测模型

直接用双方全局位置差构造相对位置测量：

`z_k = p_{g,k} - p_{a,k}`

观测方程写成：

`z_k = r_k + n_k`

其中：

`n_k ~ N(0, R_k)`

测量协方差固定构造为：

`R_k = Σ_{g,k} + Σ_{a,k}`

这里采用的工程假设是：UAV 与 UGV 的位置误差相互独立。

v1 不建模交叉协方差，也不估计相关噪声。

## 5. KF 递推在本模块里到底做了什么

### 5.1 预测

状态预测：

`r̂_{k|k-1} = r̂_{k-1|k-1} + Δt_{k-1} * u_{k-1}`

协方差预测：

`P_{k|k-1} = P_{k-1|k-1} + Q_{k-1}`

这一步在工程上表示：

- 上一拍的相对位置估计，先按相对速度推进到当前时刻；
- 同时承认“纯靠模型往前推会越来越不确定”，所以协方差增加。

### 5.2 更新

残差：

`y_k = z_k - r̂_{k|k-1}`

残差协方差：

`S_k = P_{k|k-1} + R_k`

Kalman 增益：

`K_k = P_{k|k-1} * S_k^{-1}`

状态更新：

`r̂_{k|k} = r̂_{k|k-1} + K_k * y_k`

协方差更新：

`P_{k|k} = (I_2 - K_k) * P_{k|k-1}`

### 5.3 `K_k` 的工程解释

`K_k` 决定“这一次更新到底更信预测还是更信测量”。

- 若 `R_k` 很小，说明位置测量可信，那么 `K_k` 趋向于大，更愿意跟着 `z_k` 走；
- 若 `R_k` 很大，说明位置测量不可信，那么 `K_k` 趋向于小，更愿意保留预测结果。

所以 `K_k` 不是一个“额外调参项”，而是由 `P` 和 `R` 自动计算出来的融合权重。

---

## 6. `Q / R / K` 在当前项目里的工程含义

### 6.1 `Q_k`：预测扩散，承认模型不完美

`Q_k` 吸收的误差包括但不限于：

- UAV 速度与位置时间戳不完全同步；
- UGV 速度来自车体系，需要旋转到全局平面；
- UGV 轮滑、打滑、底盘积分误差；
- UAV VIO / PX4 局部漂移；
- 速度缺测时只好退化为零输入或位姿差分；
- 测量被门控拒绝后，只能依赖预测继续传播。

### 6.2 `R_k`：测量可信度

`R_k` 直接描述“当前位置差测量到底有多不准”。

在当前项目里，它主要来源于：

- UAV `/uav/odom` 的位置协方差；
- UGV `/amcl_pose` 或 `/ugv/odom` / `/ugv/odometry/filtered` 的位置协方差；
- 协方差随时间老化后的额外膨胀；
- NaN、零方差、过小方差等异常情况下的保护下限。

### 6.3 `K_k`：自动融合权重

`K_k` 不是额外的接口，也不是额外的参数，而是滤波器在每一拍自动算出来的融合系数。

它的本质含义是：

> 如果当前位置差测量看起来很可靠，就多信测量；
> 如果当前位置差测量看起来很差，就多信预测。

---

## 7. 为什么必须做门控

如果只用“位置差测量”直接更新，那么一旦以下情况出现，滤波器会被瞬间拉飞：

- UAV 侧重定位重置，位置突然跳变；
- UGV AMCL 瞬间重匹配到错误位置；
- `/tf` 暂时异常，坐标转换结果不可信；
- 各源时间差过大，错误地把不同时间的位姿相减。

因此 v1 必须做 innovation gating。

### 7.1 门控量

马氏距离定义为：

`d_k² = y_k^T * S_k^{-1} * y_k`

### 7.2 固定阈值

二维观测，自由度为 2，固定使用：

`χ²(2, 0.95) = 5.99`

### 7.3 v1 行为

若 `d_k² > 5.99`：

- 本次测量更新直接拒绝；
- 状态维持预测结果；
- 协方差不做测量收缩；
- `gate_rejected` 计数增加；
- `Q_k` 模式切换到 `gate_rejected`，让不确定性继续保守增长。

这一步是为了让系统在“位置测量偶发跳变”时仍然稳定。

## 8. 为什么误差上界一定要写成 `γ * sqrt(λ_max(P))`

这是当前设计里最需要明确写死的一条。

二维位置协方差 `P_k` 的单位是 `m²`。
其最大特征值 `λ_max(P_k)` 的单位仍然是 `m²`。

因此标准差上界的单位应该是：

`sqrt(λ_max(P_k))`，单位 `m`

再乘以无量纲置信系数 `γ` 后，得到：

`ρ_k = γ * sqrt(λ_max(P_k))`

这才是可与“1 m 阈值”直接比较的量。

如果写成：

`ρ_k = γ * λ_max(P_k)`

那单位就变成了 `m²`，量纲错误，不能直接作为距离上界使用。

### 8.1 v1 固定用法

- `γ = 2.0`
- 进入重定位请求：`ρ_k > 1.0 m`
- 退出重定位请求：`ρ_k < 0.7 m`

并且加入迟滞计数，避免抖振。

## 9. 当前项目下的输入映射

这一部分非常关键：同一个数学模型，在当前仓库里必须被真实接口“喂起来”。

### 9.1 UAV 输入映射

#### 统一对外接口

v1 固定使用：

- `/uav/odom`：`nav_msgs/Odometry`

#### 仿真来源

当前仿真中，`tf_bridge_node` 已发布 `/uav/odom`。

#### 真机来源

当前真机方案固定要求：

- 将 `px4_planar_state_reader_node` 纳入 UAV 真机 launch；
- 配置其输出 `output_odom_topic=/uav/odom`；
- 使仿真和真机都对融合模块暴露同一个 UAV 标准接口。

#### UAV 速度语义

当前仓库中的 UAV 速度来源默认按“全局平面速度”处理：

- PX4 平面状态读取器输出的速度是 ENU / 世界系平面速度；
- 仿真 `tf_bridge_node` 当前也是由连续位姿差分得到平面速度。

因此 v1 默认将 UAV `twist.linear.x/y` 视为**非机体系速度**。

### 9.2 UGV 输入映射

#### UGV 全局位置优先级

1. 首选 `/amcl_pose`
2. 回退 `/ugv/odom`（仅在 `assume_ugv_odom_is_global=true` 时）

这里必须强调：

- `/ugv/odom` 不是一般意义上的“全局定位”；
- 它只在当前联合仿真默认场景中，由于 `ugv_map -> ugv_odom` 被静态置零，才能作为回退全局位姿使用；
- 真机导航场景下应优先使用 `/amcl_pose`。

#### UGV 速度优先级

1. 首选 `/ugv/odometry/filtered`
2. 次选 `/ugv/odom`
3. 再次回退为全局位姿差分

#### UGV 速度语义

当前仓库中，UGV odom 线速度更接近底盘 / 子体系速度，而不是严格的 `global` 平面速度。

因此 v1 固定要求：

- 使用最新全局 yaw，把 `twist.linear.x/y` 从 body / child frame 旋转到 `global` 平面；
- 如果没有可用全局 yaw，则该速度源不能直接进入预测，只能降级为位姿差分或零输入。

## 10. 为什么要分离“状态适配”和“滤波核心”

这一点是整个实现能否长期可维护的关键。

### 10.1 不分离会出什么问题

如果把下面这些都塞进一个 ROS 2 node 里：

- `/uav/odom` 订阅与字段解释；
- `/amcl_pose` 与 `/ugv/odometry/filtered` 的优先级切换；
- TF 变换；
- 速度回退与位姿差分；
- KF 预测与更新；
- 门控、诊断、触发状态机；

那么后续一旦换一个输入来源、补一个 launch、加一个回退策略，核心滤波逻辑就会被跟着改。

### 10.2 v1 固定拆分

因此 v1 固定采用以下边界：

- `AgentPlanarState`：统一内部状态契约
- `UavStateAdapter`：只负责 UAV 输入归一化
- `UgvStateAdapter`：只负责 UGV 输入归一化
- `TimeAlignmentBuffer`：只负责缓存与选取同步样本
- `RelativePositionFilter`：只做数学递推
- `RelocalizationMonitor`：只看 `P / ρ / gate status` 产生事件信号
- `DiagnosticsPublisher`：只负责对外可观测性

这样做的收益是：

- 数学核心可以直接单测；
- 输入适配可以按场景独立演化；
- 仿真与真机只需换参数与 launch，不必改滤波器本体。

## 11. 触发逻辑的工程定义

v1 不做“重定位执行器”，但必须做“重定位请求触发器”。

### 11.1 触发来源

下列情况会驱动触发器进入激活态：

1. 滤波器尚未初始化；
2. 关键输入长期失效，无法形成可信相对估计；
3. `ρ_k > 1.0 m` 持续超过进入计数阈值；
4. 连续门控拒绝次数超过阈值，且 `ρ_k` 未回落。

### 11.2 退出条件

触发器退出要求同时满足：

- 滤波器已经初始化；
- 最近若干拍存在成功更新；
- `ρ_k < 0.7 m` 持续超过退出计数阈值。

### 11.3 为什么只发布信号，不直接执行

当前仓库里没有统一的“UAV 重定位执行器”抽象。
如果 v1 直接在该模块内部调用 service / action：

- 会把估计模块与具体控制策略耦死；
- 会迫使模块知道 UAV 的飞控状态机细节；
- 会让仿真与真机的行为接口变得不一致。

因此 v1 只输出：

- `relocalize_requested = true / false`

由上层 orchestrator、行为树或人工流程决定下一步怎么做。

---

## 12. 本模块明确不负责的内容

为了防止后续实现时边界膨胀，下面这些内容明确不属于 v1：

- 不维护 `global -> uav_map` / `global -> ugv_map` 的动态对齐；
- 不修改 PX4 / UGV 底盘原有控制接口；
- 不接管视觉降落、Tag 检测或末端对接；
- 不对上层任务规划给出具体机动策略；
- 不做 3D 相对轨迹估计；
- 不负责 map 对齐失败时的全局恢复算法本身。

## 13. 架构小结

本模块的本质可以用一句话概括：

> 它是一个面向当前 DOCO 工作区的二维相对位置递推融合器：
> 先把 UAV / UGV 的异构输入统一到共享 `global` 坐标，
> 再用速度差做预测、位置差做更新、协方差做不确定性传播，
> 最后把“相对位置估计 + 误差上界 + 重定位触发”对外发布。

这也是为什么后续实现时必须坚持三条原则：

1. **输入归一化先于滤波**；
2. **滤波核心不关心具体话题来源**；
3. **触发信号与执行行为分离**。
